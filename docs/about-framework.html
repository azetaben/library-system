Library System Automation Framework

End-to-end explanation of the current framework in C:\Users\benaz\WebstormProjects\library-system, from the moment a test command starts until reports are produced.

1. What This Project Is

This repository is a TypeScript UI automation framework for the Books Inventory application. It is built mainly around Cucumber for scenario definition, Playwright for browser automation, and Allure/Cucumber HTML for reporting.

It is not a front-end or back-end application framework. Its purpose is to test an existing web application by driving a real browser and verifying behavior such as login, navigation, CRUD actions, validation, authorization, and logout.

2. High-Level Execution Flow

  1. You run npm run test:e2e.
  2. package.json starts cucumber-js through tsx.
  3. cucumber.js loads configuration, support files, utilities, and step definitions.
  4. Cucumber reads feature files from features/.
  5. Hooks start Playwright, create a fresh browser context per scenario, and build a page manager.
  6. Each Gherkin step is matched to a TypeScript step definition in e2e-tests/steps/.
  7. Step definitions call page objects in e2e-tests/pages/.
  8. Page objects interact with the Books Inventory UI through Playwright locators and actions.
  9. Assertions verify the result.
  10. Hooks save screenshots, traces, JSON, HTML, and Allure output.

3. Start Point: Package Scripts

The entry point is package.json. The main script is test:e2e, which runs: node --import tsx ./node_modules/@cucumber/cucumber/bin/cucumber-js.

This matters because the framework does not compile TypeScript into a separate build output first. Instead, tsx lets Node execute TypeScript modules directly.

4. Cucumber Configuration

The file cucumber.js tells Cucumber what to load and what to output.

  • It imports TypeScript modules from e2e-tests/config, e2e-tests/utils, e2e-tests/support, and e2e-tests/steps.
  • It writes HTML and JSON output.
  • It enables the Allure Cucumber reporter.
  • It excludes scenarios tagged with @bug.

This file is effectively the runtime wiring for the framework.

5. Environment Configuration

Runtime settings come from e2e-tests/config/env.config.ts. It loads .env and .env.example, then resolves values such as:

  • BASE_URL
  • BROWSER
  • TEST_USERNAME and TEST_PASSWORD
  • headless mode
  • slow motion
  • action, default, navigation, and Cucumber timeouts
  • viewport/download/HTTPS behavior

This module exports a single envConfig object that the rest of the framework consumes.

6. Browser Policy and Browser Creation

Browser launch policy is defined in e2e-tests/config/browser.config.ts. It sets:

  • launch arguments such as --start-maximized and --no-sandbox
  • whether the browser is headless
  • slow motion delay
  • context options such as download support and HTTPS error handling
  • the default wait strategy used by pages

The actual creation of the browser, context, and page happens in e2e-tests/utils/BrowserFactory.ts. That factory delegates browser selection to e2e-tests/helperUtilities/browserHelpers.ts, which chooses Chromium, Firefox, or WebKit based on the configured browser name.

7. Scenario Lifecycle

Scenario lifecycle is managed in e2e-tests/support/hooks.ts, while scenario state is stored in e2e-tests/support/world.ts.

Before All Scenarios

  • set the global Cucumber timeout
  • initialize Allure runtime
  • ensure report directories exist
  • launch one shared browser instance

Before Each Scenario

  • reset test data
  • create a fresh browser context
  • start Playwright tracing
  • create a new page
  • create a scenario-scoped PageManager
  • attach Allure metadata such as suite, feature, story, and parameters

After Each Scenario

  • capture a screenshot if the scenario failed
  • stop tracing and save the trace file
  • close the page and browser context

After All Scenarios

  • close the global browser

This gives the framework scenario isolation while still avoiding the cost of relaunching the whole browser for every test.

8. The World Object

CustomWorld is the shared object passed into step definitions through this. It holds:

  • the current browser
  • the current browser context
  • the current Playwright page
  • the current PageManager
  • book data created during a scenario
  • cross-step scenario state such as titles used later for verification

Without the world object, each step would be isolated and unable to share runtime objects or scenario data.

9. Feature Files

The features/ folder contains business-readable Gherkin scenarios. These are grouped by behavior areas such as:

  • authentication
  • create book
  • edit book
  • delete book
  • end-to-end CRUD
  • accessibility and performance
  • UI validation and link checks

A feature file says what the user does and what result is expected. It does not contain browser code.

10. Step Definitions

The implementation of Gherkin steps lives in e2e-tests/steps/. This is the orchestration layer of the framework.

For example:

  • commonNaviSteps.ts handles navigation, login routes, page transitions, and shared controls.
  • commonSteps.ts covers cross-form behavior and common assertions.
  • addBookSteps.ts handles the add-book workflow.
  • booksCatalogSteps.ts handles catalog visibility, row verification, pagination, and actions.

These files decide what page object method to call and what assertion helper to use. They are where feature language is translated into real browser work.

11. Page Object Layer

The UI abstraction is in e2e-tests/pages/. This framework uses page objects mainly for locators and actions.

Base Page

BasePage.ts contains common capabilities such as URL building, navigation, clicking, filling, waiting, and text retrieval. It also has a special navigateWhenResponsive method that warms an endpoint until it responds fast enough before proceeding.

Page Manager

PageManager.ts lazily creates and caches one page object per scenario. This keeps step definitions clean because they can ask for this.pm.getLoginPage() or this.pm.getBooksCatalogPage() without manually constructing objects each time.

Concrete Page Objects

  • LandingPage.ts: landing-page headings, image, and Start Testing button
  • LoginPage.ts: username/password fields, visibility toggle, login action, and error messages
  • BooksCatalogPage.ts: book rows, add/edit/delete buttons, pagination controls, and catalog row targeting
  • BookFormPage.ts: common add/edit form behavior through reusable field registration
  • AddBookPage.ts: add-book form fields, required-error access, fill-and-submit logic, select handling
  • EditBookPage.ts: edit form fields, update flow, cancel button, save action, and field clearing
  • TopNaviPage.ts: welcome message and logout control

This separation keeps low-level UI selectors inside page classes instead of spreading them across step files.

12. Assertion Strategy

A notable design choice in this framework is that assertions are mostly kept out of the page object layer. Instead, assertion helpers live next to step definitions, for example in files such as addBookAssertions.ts and booksCatalogAssertions.ts.

This means page objects answer the question “how do I interact with the page?”, while step and assertion files answer “what result should be true?”.

13. Waiting and Stability Helpers

The file e2e-tests/helperUtilities/waitHelpers.ts centralizes waiting behavior such as visible, hidden, attached, enabled, editable, page-ready, and network-idle checks.

This improves reliability by avoiding ad hoc waits inside every test step and by using one shared waiting policy across the project.

14. Test Data and Shared App Constants

Framework data is split between generated test data and stable app metadata.

  • e2e-tests/utils/TestData.ts exposes scenario data and resets it before each scenario.
  • BookFactory creates unique books for CRUD tests.
  • e2e-tests/utils/AppData.ts stores expected titles, headings, button labels, validation messages, and seeded catalog books.

This reduces duplication and makes selectors/assertions more maintainable.

15. A Complete Example Path

Consider the CRUD scenario in features/e2e/TC-12-books-crud-journey.e2e.feature. The framework processes it in this order:

  1. The scenario step text is read by Cucumber.
  2. A matching step definition in commonNaviSteps.ts, addBookSteps.ts, booksCatalogSteps.ts, or related files is executed.
  3. The step definition calls methods on LandingPage, LoginPage, BooksCatalogPage, AddBookPage, or EditBookPage.
  4. Those page objects drive the browser through Playwright.
  5. Assertions verify page title, heading, form labels, validation messages, catalog rows, and authentication state.
  6. After the scenario, trace and screenshot artifacts are saved by hooks.

16. Reporting

The framework produces several outputs:

  • cucumber-report.html for human-readable Cucumber HTML
  • e2e-tests/reports/cucumber-json/cucumber-report.json for machine-readable result data
  • allure-results/ for Allure report generation
  • screenshots and traces under e2e-tests/reports/

The script scripts/generate-multi-cucumber-report.mjs also copies the generated Cucumber HTML report into the report directory structure.

17. Role of Playwright Config

The repository also contains playwright.config.ts. It defines native Playwright Test configuration, browser projects, screenshots, and report behavior. However, in this repository it is secondary. The active test path is Cucumber-first, not Playwright-spec-first.

So Playwright is the browser engine underneath the framework, while Cucumber is the main runner on top.

18. Final Mental Model

Command layer: package.json

Runner layer: cucumber.js + Cucumber

Config layer: env.config.ts and browser.config.ts

Lifecycle layer: hooks.ts + world.ts

Behavior layer: feature files in features/

Orchestration layer: step definitions in e2e-tests/steps/

UI abstraction layer: page objects in e2e-tests/pages/

Automation engine: Playwright browser, context, page, locators, and actions

Output layer: Cucumber HTML/JSON, Allure, screenshots, and traces
